## Column Definitions

| Column | Type | Description |
|--------|------|-------------|
| sovereign_id | TEXT | FK to `cl.company_identity.sovereign_company_id` |
| outreach_id | TEXT | FK to outreach lifecycle (`cl.company_identity.outreach_id`) |
| subhub | TEXT | Always `CLI` for this sheet |
| code | VARCHAR(2) | Zero-padded movement code (01, 02, etc.) |
| cid | TEXT | Generated by `cl.mint_communication_id()` — never manual |
| movement_description | TEXT | Human-readable label for this movement code |
| cycle_state | TEXT | OPEN / EXECUTING / COMPLETED / FROZEN |
| fire_trigger | TEXT | What event fires this CID |
| fire_schedule | TEXT | When/how often this fires |
| created_at | TIMESTAMPTZ | Row creation timestamp |
| notes | TEXT | Optional context |

---

## Governance Rules

1. Codes are scoped to the `CLI` namespace — independent of OUT and SAL codes
2. Codes are additive only — never remove or repurpose a code
3. All CIDs minted via `cl.mint_communication_id(sovereign_id, outreach_id, 'CLI', code)`
4. Movement codes must exist in `cl.movement_code_registry` before minting
5. Code 05 (cycle closed) is terminal — no further CIDs should be minted in a closed cycle
6. Client CIDs should only be minted for companies that have been promoted to client (`client_id IS NOT NULL`)
7. Code 03 (quarterly check-in) is recurring — can fire multiple CIDs per cycle on a quarterly cadence
